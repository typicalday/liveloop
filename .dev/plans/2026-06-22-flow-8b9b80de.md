# Build plan: liveloop engine — terminal-settle invariant + provideInput cascade-up

## Goal

Fix two independent engine correctness defects:

1. **Issue 1 (model.ts):** once any `terminal: true` artifact in the workflow is green, the `maintainDecisions` completion-evaluator re-arm loops (allGreen and idle) must not re-arm. A terminal-green artifact is the engine's "done and frozen" signal; re-arming a completion evaluator after that point spuriously un-does a finished workflow.

2. **Issue 2 (engine.ts):** `provideInput` must call `maintainCalls` after its transaction commits, so that a human-supplied input to a parent workflow immediately cascades to any `calls:` child — matching the existing `tick` / `emit` / `green` / `close` cascade-up patterns.

Delivery: pure engine + tests, no CLI, no docs beyond doc-comments, no protocol changes, `npm run check` green on Node 22 and 24.

---

## Code orientation (what was read)

- `src/model.ts` — `maintainDecisions` (exported, pure): forward reject cascade (lines ~500-585), allGreen re-arm loop (lines 587-612), idle re-arm loop (614-633). The function named in the proposal as `maintainArts` is actually `maintainDecisions` — the only maintenance-cascade function.
- `src/engine.ts` — `provideInput` (lines 247-273): writes inside `store.tx`, calls `this.settle(workflow, def)`, fires events, then returns. It does NOT call `maintainCalls`. Compare with `tick` (line 413-), which calls `this.maintainCalls(workflow, def, now)` **outside** the tx before `store.tx(() => this.settle(...) ...)` starts.
- `maintainCalls` (lines 284-392): private, guarded by `_inMaintainCalls = new Set<string>()`. At Step 5 it calls `this.provideInput(existingChild.id, ...)` — so calling `maintainCalls` from `provideInput` introduces a call chain: parentProvide → maintainCalls(parent) → provideInput(child) → maintainCalls(child) → triggerParentIfChild(child) → maintainCalls(parent) [blocked by guard]. Safe.
- `triggerParentIfChild` (lines 399-409): called after `green` and `close` to cascade child progress to parent. `provideInput` does not call it and does not need to — it is called on the parent directly.
- Test files: `test/evaluator.test.ts` (allGreen trigger, pure-model tests using `arts()` / `def()` / `loop()` from `test/helpers.ts`), `test/calls.test.ts` (engine integration tests using `makeEngine`, `openStore(':memory:')`).
- `test/helpers.ts` — `arts()` builder supports `terminal` on artifact specs. The `loop()` builder supports `terminal` on loop specs.

## Assumptions

- "Terminal artifact" means `ArtifactData.terminal === true`, derived from `LoopDef.terminal === true` at green time. This is a stored field on the artifact, discoverable by iterating `arts.values()` — no def lookup needed.
- The guard condition is: `[...arts.values()].some(a => a.acceptance === 'green' && a.terminal === true)`. A helper can be extracted inside `maintainDecisions` for clarity.
- Idle re-arm also gets the same guard (proposal confirms both branches are symmetric).
- For Issue 2, `now` is not passed to `maintainCalls` from `provideInput` (matching the pattern in `triggerParentIfChild` at line 407 which also omits `now`). The `now` parameter on `maintainCalls` is optional; not passing it is correct — a human provide is not a tick and the idle clock path is irrelevant.

## Implementation: Issue 2 — src/engine.ts

### Where

`provideInput`, after the existing `this.fireSettled(workflow)` call at line 272.

### What to add

```typescript
    // M2B cascade-up: if this workflow has calls: loops, re-run child maintenance so
    // a newly-provided input is immediately re-provided to any child that maps from it.
    // Must run outside the tx (maintainCalls opens its own store writes).
    // No `now` — a human provide is not a tick; idle timing is irrelevant here.
    this.maintainCalls(workflow, def);
```

### Why this is safe

- `maintainCalls` is guarded by `_inMaintainCalls`; if `provideInput` was itself called from within `maintainCalls` (Step 5, child re-provide), the parent's id is already in the set, so the outer `maintainCalls(parentWf)` guard in `triggerParentIfChild` blocks re-entry. The new call here is `maintainCalls(workflow, def)` where `workflow` is the parent that received a direct human input — a different workflow id from any child currently inside `_inMaintainCalls`.
- The call is outside any store tx — same as `tick`'s call at line 417 and `triggerParentIfChild`'s call at line 407.
- For workflows with no `calls:` loops, `maintainCalls` iterates the loops and finds nothing matching `loop.calls` — exits cleanly with no writes.

### Exact edit

After line 272 (`this.fireSettled(workflow);`), insert the block above. The full method body after the edit:

```typescript
  provideInput(workflow: string, name: string, value: Record<string, unknown>): void {
    const def = this.defFor(workflow);
    const inputDef = def.inputs.find((i) => i.name === name);
    this.store.tx(() => {
      const art = this.store.getArtifact(workflow, name);
      if (!art) throw new Error(`no such input artifact: ${name}`);
      if (inputDef?.schema !== undefined) {
        const check = validateValue(inputDef.schema, value);
        if (!check.valid) {
          throw new Error(`input '${name}' failed schema: ${summarizeIssues(check.issues)}`);
        }
      }
      this.store.putArtifact({
        ...art,
        acceptance: 'green',
        version: art.version + 1,
        value,
      });
      this.settle(workflow, def);
    });
    this.fire({ type: 'commit', workflow, path: name, action: 'provide' });
    this.fireSettled(workflow);
    // M2B cascade-up: re-run calls: child maintenance so a newly-provided input
    // is immediately re-provided to any child that maps from it (no extra tick needed).
    // Outside the tx — same contract as tick's maintainCalls call at line 417.
    this.maintainCalls(workflow, def);
  }
```

## Tests: Issue 1 — test/evaluator.test.ts

Add two new tests after the existing test `(e)`. Label them `(f)` and `(g)` to continue the sequence.

### Test (f) — regression: terminal-green seals the workflow, evaluator NOT re-armed

```
test('(f) terminal-settle invariant: evaluator not re-armed when terminal artifact is green', () => {
  // Workflow: planner → plan, merger (terminal) → merge (terminal:true), completion (allGreen) → outcome
  // Setup: all green including merge (terminal), outcome green → workflow done
  // Inject a debt: plan goes back to owed (simulates mid-pipeline re-arm after terminal green)
  // maintainDecisions: outcome must NOT be re-armed (terminal artifact is green)

  import { maintainDecisions } from '../src/model.ts';  // already imported
  const d = def('terminal-wf', [input('proposal')], [
    loop({ name: 'planner', consumes: ['proposal'], produces: ['plan'] }),
    loop({ name: 'merger', consumes: ['plan'], produces: ['merge'], terminal: true }),
    loop({ name: 'completion', produces: ['outcome'], on: ['allGreen'] }),
  ]);

  // State: merge is green+terminal, plan went back to owed (mid-pipeline debt after terminal)
  const state = arts([
    { path: 'proposal', producer: 'human', acceptance: 'green', version: 2 },
    { path: 'plan',     producer: 'planner', acceptance: 'owed', version: 1 },
    { path: 'merge',    producer: 'merger',  acceptance: 'green', version: 1, terminal: true },
    { path: 'outcome',  producer: 'completion', acceptance: 'green', version: 1 },
  ]);

  const ops = maintainDecisions(d, state);
  const rearmOps = ops.filter((op) => op.kind === 'reject' && op.path === 'outcome');
  assert.equal(rearmOps.length, 0, 'outcome must NOT be re-armed when a terminal artifact is green');
});
```

### Test (g) — non-regression: poller (no terminal) still re-arms normally

```
test('(g) non-regression: poller (no terminal artifact) still re-arms when workflow falls out of done', () => {
  // Same shape as test (d) but explicitly checking the no-terminal path is unaffected
  const d = def('poller-wf', [input('proposal')], [
    loop({ name: 'planner', consumes: ['proposal'], produces: ['plan'] }),
    loop({ name: 'completion', produces: ['outcome'], on: ['allGreen'] }),
  ]);
  // No terminal artifact — plan is owed after a fall-out-of-done
  const state = arts([
    { path: 'proposal', producer: 'human', acceptance: 'green', version: 2 },
    { path: 'plan',     producer: 'planner', acceptance: 'owed', version: 1 },
    { path: 'outcome',  producer: 'completion', acceptance: 'green', version: 1 },
  ]);
  const ops = maintainDecisions(d, state);
  const rearmOps = ops.filter((op) => op.kind === 'reject' && op.path === 'outcome');
  assert.equal(rearmOps.length, 1, 'outcome MUST be re-armed when no terminal artifact is green (poller pattern)');
});
```

### Test (g) — regression: provideInput on parent cascades to child without extra tick

Use the existing `childDef` and `parentDef` fixtures already in the file. The test scenario:

1. Create parent, tick → provision order issued.
2. Green sandbox → child spawned.
3. Tick parent → child exists; child's `data` has `{env: 'v1'}` (from sandbox).
4. Tick child → worker order. (Do NOT green it yet — leave child in flight / owed-order state.)
5. **Re-provide sandbox directly on the parent** using `engine.provideInput(parentWf, 'sandbox', {env:'v2'})` — this is the human/external re-provide path, bypassing the worker.
6. Assert: WITHOUT any additional tick on the parent, the child's `data` artifact value is `{env: 'v2'}`. This is the cascade that `provideInput` must trigger immediately.

If the fix is absent, step 6 fails (child still has `{env: 'v1'}`); after the fix, it passes.

**Important note on the scenario:** `provideInput` in the current test harness calls `provideInput(parentWf, 'sandbox', ...)`. But `sandbox` is NOT an `input:` of `parentDef` — it is an artifact produced by the `provision` loop. Check whether `provideInput` validates that the path must be a declared `def.inputs` entry.

Looking at the code: `this.store.getArtifact(workflow, name)` will find any artifact, and `def.inputs.find((i) => i.name === name)` may return undefined (no schema to check) — the artifact is still written green. So `provideInput` can be called on any existing artifact path, not just declared inputs. But to make the test clean and representative, use `engine.provideInput` on a declared seedOwed input of the parent. The cleanest approach: create a parent def with a **second declared input** that feeds into `callsInputs` so that the human can re-provide it and the cascade must happen.

Alternative: create a simpler fixture for this specific test — a `parentDirect` def where:
- `input: data2` (seedOwed) feeds directly into `calls: callsInputs: {data: 'data2'}`.
- `provideInput(parentWf, 'data2', {env:'v2'})` is the human provide.
- The test checks that after `provideInput`, child `data` is `{env:'v2'}` without an extra tick.

This keeps the test self-contained and avoids depending on `provideInput` working on non-input artifacts. The builder should choose whichever approach cleanly validates the behaviour, but the simpler fixture approach is preferred.

  // Create parent with data2=v1 provided

## Edge cases and constraints

1. **Terminal-settle guard must not affect non-terminal workflows.** The `anyTerminalGreen` variable is always computed but only gates the re-arm path. For a workflow with zero terminal artifacts, `anyTerminalGreen` is always `false`, so existing behaviour is unchanged.

2. **Idle guard placement matters.** The idle guard must be placed AFTER `if (!idleEligible(loop, arts, time)) continue;` so we don't compute `anyTerminalGreen` unnecessarily when the alarm hasn't elapsed. Actually `anyTerminalGreen` is computed once outside both loops so placement within the loop is fine — just add the `if (anyTerminalGreen) continue;` after the `idleEligible` check to minimise confusion about execution order.

3. **No `anyTerminalGreen` in the forward cascade.** The forward cascade (lines 500-585) already handles terminal artifacts correctly (`if (art.acceptance === 'green' && !art.terminal)` skips terminals). Do not touch those lines.

4. **provideInput on non-parent workflows.** Calling `maintainCalls(workflow, def)` on a workflow with no `calls:` loops is a no-op (the for loop finds no `loop.calls`). No regression risk.

5. **Nested provideInput during maintainCalls.** When `maintainCalls` Step 5 calls `provideInput(childId, ...)`, that child's `provideInput` calls `maintainCalls(childId, childDef)`. The child is not in `_inMaintainCalls` (only the parent is), so this is fine. The child's `maintainCalls` will then call `triggerParentIfChild(childId)` which calls `maintainCalls(parentWf, ...)` — BLOCKED by `_inMaintainCalls` since the parent is still inside its own `maintainCalls`. Guard holds.

6. **`now` omission.** Not passing `now` to `maintainCalls` from `provideInput` matches `triggerParentIfChild` at line 407. The `now` parameter only matters for idle-trigger cadence inside `settle`; human provides don't have a clock context for that.

7. **Existing tests.** All 424 existing tests must continue to pass. The changes are additive guards, not behavioural removals.
